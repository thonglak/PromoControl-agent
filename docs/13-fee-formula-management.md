# Fee Formula Management — จัดการสูตรคำนวณค่าธรรมเนียม

## ภาพรวม

ระบบจัดการสูตรคำนวณสำหรับของแถมที่มี `value_mode = 'calculated'` ประกอบด้วย 3 หน้า:

```
Fee Formula Management
├── 1. หน้าจัดการสูตร (Formula Management)
│      CRUD fee_formulas ผูกกับ promotion_item_master
├── 2. หน้าจัดการมาตรการ/นโยบาย (Policy Management)
│      CRUD fee_rate_policies ผูกกับ fee_formulas
└── 3. หน้าทดสอบสูตร (Formula Tester)
│      จำลองคำนวณด้วยข้อมูลจริงหรือสมมติ
```

---

## Menu Structure

```
Sidebar:
├── ...
├── Promotion Items         → /promotion-items
├── Fee Formulas            → /fee-formulas          ← ใหม่
│   ├── สูตรคำนวณ           → /fee-formulas
│   ├── มาตรการ/นโยบาย      → /fee-formulas/policies
│   └── ทดสอบสูตร           → /fee-formulas/tester
├── Sales Entry             → /sales-entry
├── ...
```

**สิทธิ์การเข้าถึง:**
- `admin`, `manager` → เข้าถึงทุกหน้า (CRUD)
- `sales` → เข้าถึงได้เฉพาะหน้าทดสอบสูตร (read-only)
- `finance`, `viewer` → ดูอย่างเดียว (read-only)

- สูตรแสดงเป็นภาษาไทย เช่น "ราคาประเมิน × 2.00% × 50%"

### Validation Rules (Create/Edit)

| ฟิลด์ | กฎ |
|-------|-----|
| รายการของแถม | required; ต้องเป็นรายการที่ `value_mode = 'calculated'`; ต้องไม่มีสูตรผูกอยู่แล้ว (ยกเว้นกรณีแก้ไข) |
| ฐานคำนวณ | required; enum |
| ชื่อช่องกรอก | required เมื่อ base_field = 'manual_input' |
| อัตราปกติ | required; > 0; ≤ 100 |
| สัดส่วนผู้ซื้อ | required; > 0; ≤ 100 |
| คำอธิบาย | optional; max 200 chars |

### Delete Rules

- ถ้ามี `fee_rate_policies` ผูกอยู่ → ต้องลบ policies ก่อน (หรือ cascade delete with confirmation)
- ถ้ามี `sales_transaction_items` ที่ใช้สูตรนี้คำนวณแล้ว → **ไม่สามารถลบได้** แสดง warning "สูตรนี้ถูกใช้ในการบันทึกขายแล้ว"
- Confirmation dialog: "ยืนยันลบสูตร 'ค่าธรรมเนียมโอน'? นโยบายที่ผูกอยู่ X รายการจะถูกลบไปด้วย"

### การเชื่อมกับ Promotion Item Master

เมื่อสร้างสูตรสำเร็จ:
1. ถ้า `promotion_item_master.value_mode` ยังไม่เป็น `calculated` → อัปเดตเป็น `calculated` อัตโนมัติ
2. ถ้าลบสูตรสำเร็จ → อัปเดต `value_mode` กลับเป็น `fixed` (หรือ `manual`) พร้อม warning

| ลำดับ | `priority` | ตัวเลข |
| สถานะ | chip | Active (เขียว) / Expired (เทา) / Upcoming (ฟ้า) / Inactive (แดง) |
| Actions | แก้ไข, ลบ, Toggle active | icon buttons |

**สถานะ (คำนวณจาก is_active + วันที่):**

| เงื่อนไข | สถานะ | สี |
|----------|-------|-----|
| `is_active = true` AND `effective_from ≤ today ≤ effective_to` | Active | เขียว |
| `is_active = true` AND `effective_from > today` | Upcoming | ฟ้า |
| `is_active = true` AND `effective_to < today` | Expired | เทา |
| `is_active = false` | Inactive | แดง |

### Conditions Builder

เงื่อนไขแสดงเป็น checkbox + input ที่ user-friendly ไม่ต้องเขียน JSON เอง:

| Condition | UI | JSON Output |
|-----------|-----|-------------|
| จำกัดราคาขาย | checkbox + number input | `{"max_base_price": 3000000}` |
| จำกัดประเภทโครงการ | checkbox + chip list | `{"project_types": ["condo","house"]}` |

> **หมายเหตุ:** เพิ่มเงื่อนไขใหม่ได้ในอนาคตโดยเพิ่ม UI component + update `evaluateConditions()` ใน frontend

| ฟิลด์ | กฎ |

| สัดส่วนผู้ซื้อใหม่ | optional (nullable); ถ้ากรอก: > 0; ≤ 100 |

### ฟีเจอร์เพิ่มเติม

**1. Batch Test (ทดสอบหลายยูนิตพร้อมกัน):**
- ปุ่ม "ทดสอบทุกยูนิตในโครงการ" → คำนวณสูตรสำหรับทุกยูนิตที่ status = available/reserved
- แสดงผลเป็นตาราง: ยูนิต | ราคาขาย | ราคาประเมิน | ค่าโอน | ค่าจดจำนอง | รวม | นโยบาย
- Export เป็น Excel ได้

**2. สรุปผลกระทบงบ (Budget Impact):**
- แสดงว่าถ้าทุกยูนิตในโครงการใช้สูตรนี้ จะมีผลกระทบต่องบรวมเท่าไร
- เช่น "ยูนิตที่ได้ส่วนลดจากมาตรการ: 45/50 ยูนิต | งบที่ประหยัดได้: 675,000 บาท"

## Validation & Business Rules

### สูตร (fee_formulas)

1. 1 รายการของแถม : 1 สูตร (UNIQUE constraint)
2. สูตรสร้างได้เฉพาะกับรายการที่ `value_mode = 'calculated'`
3. สร้างสูตรแล้ว → auto-update `promotion_item_master.value_mode = 'calculated'`
4. ลบสูตรแล้ว → auto-update `promotion_item_master.value_mode = 'fixed'` (พร้อม confirmation)
5. แก้ไขสูตรจะ **ไม่** กระทบ transaction ที่บันทึกไปแล้ว (used_value ถูก snapshot ไว้แล้ว)
6. `default_rate` และ `buyer_share` เก็บเป็น decimal (0.02 = 2%) แต่ UI แสดงเป็น %

### นโยบาย (fee_rate_policies)

1. 1 สูตร : N นโยบาย ได้
2. ถ้ามีหลายนโยบายตรงเงื่อนไขพร้อมกัน → ใช้ `priority` สูงสุด
3. ถ้า priority เท่ากัน → ใช้ `effective_from` ล่าสุด (นโยบายใหม่กว่า)
4. นโยบายที่หมดอายุ (expired) ยังเก็บไว้ในระบบ (ไม่ลบอัตโนมัติ) เพื่อ reference
5. `override_buyer_share = null` → ใช้ `buyer_share` จากสูตรเดิม
6. `conditions = {}` → ไม่มีเงื่อนไข ใช้กับทุกยูนิต
7. แก้ไขนโยบายจะ **ไม่** กระทบ transaction ที่บันทึกไปแล้ว

### ทดสอบสูตร

1. การทดสอบเป็น read-only ไม่สร้างข้อมูลใดๆ
2. แสดง **ทุก policy** ที่ตรวจ ไม่ว่าจะ match หรือไม่ พร้อมเหตุผล
3. แสดงค่า "ปกติ" (ไม่มี policy) vs "หลัง policy" เพื่อเปรียบเทียบ
4. Batch test จำกัด 500 ยูนิตต่อครั้ง
